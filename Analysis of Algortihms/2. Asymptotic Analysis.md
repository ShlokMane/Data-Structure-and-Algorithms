# Asymptotic Analysis

It is the idea of measuring the order of growth, it does not depend on machine, programming language etc. Also we do not
need to implement the algorithm to analyse it. So the idea is to measure the order of growth of the time taken by a
function in terms of input size. For example if the input is n so we check if the program grows linearly, quadratically
or constant etc.

Let us analysis the code written in [Analysis Of Algorithms](1.%20Analysis%20Of%20Algorithms%20(Background).md)

If you look at `fun1()` we are doing three operations for any given n i.e. multiplication, addition and division. And we
assume that any mathematically or comparison operation takes constant time and hence `fun1()` takes constant time, let's
say C<sub>1</sub>

Now let's look at `fun2()` here we are doing a constant work in creating variable res, and then we are doing a loop that
will run n times and will do a constant work of adding res with current i. So we can say it will take C<sub>2</sub>N +
C<sub>3</sub>.

For `fun3()` we are doing a for loop inside a for loop, the inside for loops runs 1 time for i = 1, 2 times for i = 2, 3
times for i = 3 ... n times for i = n so we can say it runs n*(n+1)/2 times. Also, it has some other constants for
creating res variable. So we can say it is C<sub>4</sub>N<sup>2</sup> + C<sub>5</sub>N + C<sub>6</sub>

In order of growth we can always ignore the lower value, the idea is simple for a very huge value of N these constant
and other values don't make a difference.

![`Order Of Growth`](./images/Order%20Of%20Growth.png)

In the above image we have assumed `fun1()` ran on a slower system and hence C<sub>1</sub> is 100, whereas `fun2()` ran
on a faster system where C<sub>2</sub> and C<sub>3</sub> are 1. But even with this we see that for n greater than 999
`fun2()` will take more time than `fun1()`.
